# Cache hypothesis A/B — decision report

Generated by `scripts/cache_ab_test.py` against `floppy_simpl3.cil-2`. Three
modes per (function, property):
- **Mode 1** — uncached single-turn (baseline; current prototype behavior)
- **Mode 2** — cached single-turn (`cache_control: ephemeral` on system + source-prep)
- **Mode 3** — cached multi-turn (3 turns: enumerate → derive → emit JSON)

## Quality / equivalence

Surface-string jaccard on `requires` / `ensures` is a noisy metric — same
fact phrased two ways scores 0.0. The **content inspection** is more
informative. Pattern across both backends:

### Sonnet — mode 2 vs mode 1 content diffs

| function | property | observation |
|---|---|---|
| `FloppyPnpComplete` | memsafe | **Mode 1 fabricates `Context != NULL`; mode 2 correctly drops it.** This is the design-doc canary that triggered the redesign. Mode 2 also adds `result == -1073741802` (STATUS_INSUFFICIENT_RESOURCES). |
| `KeSetEvent` | memsafe | Mode 2 catches `setEventCalled == 1` ensure (mode 1 missed). |
| `stubMoreProcessingRequired` | memsafe | Mode 2 catches `s == NP` precondition + `s == MPR1` postcondition (mode 1 missed). |
| `FlQueueIrpToThread` | memsafe | Mode 2 enumerates 4 return values; mode 1 enumerates 3. |
| `FlQueueIrpToThread` | overflow | Mode 1 publishes unverifiable precondition on internal field; mode 2 (correctly) drops it. |

Mode 2 systematically **improves** content over mode 1 — likely because
splitting `[source-prep][question]` into two user blocks separates context
from instruction more clearly than mode 1's one-big-message style.

### llama.cpp — mode 2 vs mode 1 content diffs (KeSetEvent)

| property | observation |
|---|---|
| memsafe | Mode 2 catches `setEventCalled == 1` ensure (mode 1 missed). |
| memleak | Mode 2 drops trivial `no resource acquired` ensure — slightly less informative but not wrong. |
| overflow | Mode 2 spells out `[-2147483648, 2147483647]` vs mode 1's `[INT_MIN, INT_MAX]` — semantically identical. |

## Decision (per plan Step 6 rule)

**Mode 3 (cached multi-turn) — REJECTED.**
- Slower on both backends (+266% Sonnet, +119% llama.cpp).
- No quality benefit beyond what mode 2 delivers.
- More API roundtrips → more failure surface.
- → **Skip plan Step 7 entirely. Do not build the multi-turn Q&A driver.**

**Mode 2 (cached single-turn) — CONDITIONAL ADOPT.**
- Anthropic backend: −65% billed cost, +47% wall time. Worth it for batch
  summarization (cost matters; latency does not — bottom-up driver runs
  asynchronously). Plus content quality is better. → **Default on Claude.**
- llama.cpp backend: no $ savings, +34% wall time, no clear quality win.
  → **Default to mode 1 on local backends.**

## Implication for the contract pipeline (plan Step 4)

`ContractPass` should:

1. Send the SYSTEM_PROMPT once per session as a `system=` block with
   `cache_control: ephemeral` (Anthropic only — no-op elsewhere).
2. For each function, build the source-prep block once and send it as a
   user content block with `cache_control: ephemeral`. The three
   per-property questions then attach as a second content block (one
   request per property; the source-prep cache hit serves all three).
3. Backend selection: detect whether the backend supports ephemeral cache
   markers; if not, fall back to a single concatenated user message
   (mode-1-equivalent).
4. **No multi-turn driver.** `complete_with_metadata` with cache hints is
   the only call shape we need.

This collapses plan Steps 6, 7 into a single decision: **ship mode 2 as
the contract pass's default call shape, with a backend-driven fallback to
mode 1 when caching is not supported.**

## Caveats

- 4 functions on one project — small sample. Re-run on libpng once the
  contract pass lands to confirm the cost/wall trade scales.
- Sonnet was tested on Vertex AI `rawPredict`. Direct Anthropic API may
  show slightly different cache_creation accounting.
- The +47% Sonnet wall-time penalty is dominated by the cache-write call.
  Across many functions sharing one SYSTEM_PROMPT cache, the per-call
  wall-time gap should narrow. Re-measure on a 50+ function run.
